Architecture & Implementation Summary

Phase 1 Features Implemented

RentEase Phase 1 focuses on establishing a robust authentication system and RESTful API foundation. All features have been implemented using best practices and industry standards:

User Registration Endpoint

POST /api/v1/auth/register

  • Accepts required fields: firstname, lastname, email, password, role (RENTER or OWNER)
  • Validates input data with proper error responses (email format, password requirements, required fields)
  • Prevents duplicate email registration with 409 Conflict response code
  • Encrypts passwords using BCrypt hashing (10 rounds minimum)
  • Persists user data in Supabase PostgreSQL database
  • Returns JWT access and refresh tokens for immediate authenticated session access
  • Returns user profile (without password) in response body

User Login Endpoint

POST /api/v1/auth/login

  • Accepts email and password credentials for authentication
  • Validates credentials against database with BCrypt comparison
  • Rejects invalid credentials with 401 Unauthorized response
  • Returns JWT access token for API authentication
  • Provides secure session management across web and mobile clients
  • Includes refresh token mechanism for long-lived sessions

What Changed During Implementation

  • Files organized from technical-layer folders into feature-based folders (auth, listings, requests, etc.)
  • Package declarations updated in all Java files to match new folder structure
  • Import statements updated across all files to reference new package locations

  • Exception handling centralized in shared/exception folder

What Did NOT Change

  • All business logic and method implementations unchanged

  • All UI components and user-facing behavior unchanged
  • Authentication flows (email/password) unchanged
